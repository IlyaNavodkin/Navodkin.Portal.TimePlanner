# Требования к приложению планирования таймлайнов

## 1. Цель
Приложение нужно для планирования работ руководителем по подчиненным сотрудникам в разрезе:
- проекта;
- чарджа внутри проекта;
- календаря (по дням).

Важно: проекты и чарджи являются внешними сущностями другого микросервиса. В нашей БД они не хранятся как отдельные таблицы, сохраняются только ссылки (внешние ID).

## 2. Роли
- Руководитель: выбирает себя в селекте и планирует подчиненных.
- Система: хранит планы, отдает данные для отображения, валидирует связи через внешний сервис.

## 3. Основные сущности
- `Manager` (руководитель, приходит из внешнего микросервиса).
- `Employee` (сотрудник, приходит из внешнего микросервиса и связан с руководителем).
- `Timeline` (сущность назначения: ссылки на проект/чардж + сотрудник + метаданные).
- `TimelineDay` (один день внутри `Timeline`).

## 4. Функциональные требования
1. Выбор руководителя в верхнем селекте.
2. Фильтр проектов: мультиселект + поиск (данные из внешнего микросервиса).
3. Фильтр чарджей: мультиселект + поиск (данные из внешнего микросервиса).
4. Отображение календарного графика от текущей даты (горизонтальный скролл).
5. Отображение строк по иерархии: проект -> чарджи (по данным внешнего сервиса).
6. Добавление таймлайна для чарджа:
   - клик по ячейке/строке чарджа;
   - открытие popover;
   - поиск сотрудника;
   - скролл списка сотрудников;
   - выбор сотрудника;
   - выбор дней (или диапазона в UI с разворачиванием в дни на backend).
7. Отображение таймлайнов как прямоугольных блоков на сетке.
8. Редактирование таймлайна (дни, сотрудник, удаление).
9. Сохранение в PostgreSQL:
   - одна сущность `timeline`;
   - набор записей `timeline_days` (по 1 записи на каждый день);
   - ссылки на внешние сущности: `project_external_id`, `charge_external_id`, `manager_external_id`, `employee_external_id`.
10. Повторное восстановление экрана из БД с учетом фильтров.

## 5. Правила валидации
1. Перед созданием `Timeline` проверять через внешний сервис, что `charge_external_id` принадлежит `project_external_id`.
2. Нельзя назначить сотрудника, который не входит в команду выбранного руководителя (проверка через внешний сервис).
3. В `timeline_days` должны быть уникальные дни в рамках одного `timeline`.
4. Желательно блокировать конфликты по сотруднику на один и тот же день (настраиваемое правило).

## 6. Нефункциональные требования
- Стек: Nuxt (TypeScript), Tailwind, Node.js backend, PostgreSQL.
- Доступ к БД только через `pg` (`^8.13.3`).
- Поднятие через Docker Compose и `.env`.
- Внешний микросервис сотрудников/руководителей/проектов/чарджей можно мокать HTTP API.
- Время ответа списка/графика: целевой до 500 мс при типовой нагрузке.

## 7. User flow
1. Пользователь открывает экран планирования.
2. Выбирает руководителя.
3. Выбирает фильтры по проектам и/или чарджам.
4. Система рисует строки графика и загружает текущие таймлайны.
5. Пользователь кликает на строку чарджа и добавляет таймлайн через popover.
6. Система сохраняет `timeline` и `timeline_days`, затем сразу отрисовывает блок на графике.
